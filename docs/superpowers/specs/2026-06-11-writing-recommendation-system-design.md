# 《第九条路》写作自动推荐系统设计

## 目标

为《第九条路》建立一套可迭代的写作推荐系统，让 agent 在创作新章节、审稿修改、沉淀人物和更新伏笔时，能够基于已有文本证据自动给出稳定推荐，而不是临场凭感觉决策。

## 设计原则

1. **证据优先**：推荐必须引用项目内已有设定、章节卡、正文、人物账本或伏笔账本。
2. **连续性优先于新奇**：先解决剧情债务、人物弧线和关系变化，再追求新场景和新冲突。
3. **推荐与执行分离**：低置信度推荐只提出方向，不直接进入正文创作。
4. **事实与可能性分离**：只有已经写进正文的内容才能进入“已发生事实”；候选方向只能进入导航或推荐区。
5. **可反写**：每章定稿后，必须把人物、关系、伏笔和章节债务反写到账本，供下一章使用。

## 文件架构

- `novel/character-continuity.md`：人物连续性账本，记录人物不可违背的核心、当前状态、变化轨迹和行为证据。
- `novel/chapter-ledger.md`：章节账本，记录每章发生了什么、主角做了什么选择、付出什么代价、留下什么债务。
- `novel/foreshadowing-ledger.md`：伏笔账本，记录物件、能力异样、关系暗线和现实压力的投放与回收状态。
- `novel/story-control-checklist.md`：故事控制清单，定义 agent 自动推荐、信息充足检查、打分修改和人工确认闸门。
- `novel/chapter-navigators/README.md`：章节导航图说明。
- `novel/chapter-navigators/template.md`：每章开写前复制使用的导航模板。
- `AGENTS.md`：记录 agent 必须读取这些账本，并按推荐合约工作。

## 推荐合约

每条自动推荐必须包含以下字段：

```text
推荐动作：
依据证据：
解决的问题：
预期收益：
潜在风险：
置信度：
是否允许自动执行：
```

如果不能写出“依据证据”，该推荐不能进入正文创作。

## 四类推荐机制

### 1. 新章信息是否足够

agent 必须检查：

- 本章功能
- 现实事件
- 主角关键选择
- 选择代价
- 家庭压力数字
- 未来分支触发点
- 父母行动示范
- 章尾未完成动作或问题

缺 1-2 项时，agent 自动补推荐；缺 3 项以上时，不写正文，只输出需要补齐的信息。

### 2. 新章写什么

推荐排序：

```text
剧情债务 > 人物弧线 > 关系变化 > 伏笔回收 > 新鲜场景
```

每次最多给 3 个候选方向，并明确主推荐。主推荐必须说明为什么比另外两个更稳。

### 3. 写完后怎么打分修改

审稿评分权重：

```text
连续性安全 30%
章节推进 25%
爆款钩子 20%
人物真实 15%
语言质感 10%
```

修改排序：

```text
高提分 + 低破坏风险 > 高提分 + 高风险 > 低提分细节
```

涉及人物核心、能力规则、主线方向的修改，必须人工确认。

### 4. 写完后怎么沉淀

每章定稿后，agent 必须反写：

- 已发生事实
- 人物状态变化
- 关系状态变化
- 伏笔新增、推进、回收或废弃
- 下一章必须继承的债务
- 新增风格规则或反例

## 置信度规则

- `85%` 及以上：允许自动生成正文草案或执行低风险改稿。
- `70%-84%`：允许给出主推荐，但需要用户或主 agent 确认后执行。
- `70%` 以下：不进入正文，只补信息、列风险或请求确认。

强制人工确认项：

- 修改人物不可违背核心。
- 修改轻科幻能力规则。
- 改变九个人生节点顺序。
- 让父母从行动影响变成说教推动。
- 引入会改变作品类型的设定。

## 模型与代理分工

- 正式正文写作：`deepseek-v4-pro` 非思考模式。
- 审稿、结构诊断、伏笔推演：`deepseek-v4-pro` 思考模式，`reasoning_effort=high`。
- agent 本地推荐：先基于账本生成结构化推荐，再决定是否调用 DeepSeek 复核。
- 子代理：用于并行审查模板结构、连续性风险和计划遗漏，不直接拥有最终叙事决策权。

## 验证方式

本系统落地后，至少检查：

1. `AGENTS.md` 是否要求读取连续性账本、章节账本、伏笔账本和章节导航。
2. 每个账本是否区分“事实”“变化”“候选方向”。
3. `story-control-checklist.md` 是否包含四类推荐和人工确认闸门。
4. 章节导航模板是否能在开写前判断信息是否充足。
5. 文件中不包含 `.env` 或 API key。
